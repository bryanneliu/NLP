Pre-Experiment Design

Experiment Information

WLBR SIM link: https://sim.amazon.com/issues/WLBR-34416

Additional Helpful Links (optional / as applicable):
Project Documentation: Amazon product search specific tokenizer for English Sourcing signals
Related Experiments: SPONSORED_PRODUCTS_563415 (Relax broad match with synonyms): 14-day raw APT T2 shows 13.5M (0.07%) SP revenue gain, 0.03% click gain, 82M (0.02%) OPS gain, 88M (0.03%) GCCP gain, 21.6M (0.04%) NetCP. It’s a launchable candidate.

Objective

What are you trying to do? How do you think it will benefit shoppers, advertisers and/or Amazon? Which key segments will be impacted by this weblab (e.g. Prime, Books, Softlines, Top Spending Advertisers, Top X queries etc.)?

In SP sourcing, we mainly rely on lexical match for <query, ASIN> retrieval and <query, targeting clause keyword> matching. The tokenizer is responsible for segmenting and normalizing the text. It is the key component of the matching. Currently, we use the Lucene tokenizer, which is a simple and generic solution. However, it doesn't perform optimally for Amazon product search, resulting in numerous tokenization and matching issues. Some examples are provided below ('!=' indicates not matched). 

  1. Num+unit
    • ["10ft", "10foot", "10 foot", "10 feet", "10feet", "10'"] can not be segmented and normalized to "10 feet"
    • "10 watt 1 ohm resistor" != "10watt 1 ohm resistor"
    • "viotek 27in curved monitor" != "viotek 27 inch curved monitor"
  2. Size
    • "20 x 10-8" != "20x10-8"
    • "12 by 16 white frame" != "12x16 white frame"

    • "plastic outdoor rugs 9 x 12" != "plastic outdoor rugs 9x12"

  3. Age
    • "birthday card grandson 5 years old" != "birthday card for grandson age 5"
    • "toys for 8 years boy" != "toys for 8 year old boy"
    • "3 year old crafts" != "3 yr old crafts"
  4. Apostrophe
    • "70’s pants" != "70s pants"
    • "painter's tape 1.5 inch" != "painters tape 1.5 inch"
  5. Diacritics
    • "Sofá" != "sofa"
    • "bebé" != "bebe"
  6. Lemmatization
    • "swimming belt" != "swim belt"
    • "shredded claws" != "shredding claws"
    • "heated pad neck and shoulder" != "heat pad neck and shoulder"
  7. Num Range
    • Broad match keyword: "gift for boys age 8-12", query: "11 year old boy gift ideas", not a broad match
    • Broad match keyword: "gifts for 10+ year old boys", query: "boys gifts age 13-15 years old", not a broad match
    • Broad match keyword: "pokemon shirt boys size 8-12", query: "pokemon long sleeve shirts for boys size 8-10", not a broad match

The LATAC analyzer is used in Amazon organic search for query and search index analysis. It supports text analysis in multiple languages and incorporates data enhancements such as synonyms and compounding. To enhance matching, we conducted several experiments and analyses using the LATAC analyzer, resulting in modest increases of 0.27% to 0.49% in matches. Further analysis revealed that LATAC indeed improves many cases but also introduces certain regressions. 

We have developed the Pinpoint Token Match Engine to address tokenization and normalization issues mentioned earlier. This solution covers several areas (attributes or techniques) where performance has been lacking. It is designed to enhance match precision and recall, bringing more products to shoppers, increasing the visibility of advertiser products, and ultimately benefiting the Amazon ecosystem.

We have 2 experiment arms:

The Pinpoint Token Match Engine, based on rules, boasting ~100% accuracy, is capable of enhancing both <query, ASIN> and <query, targeting clause keyword> matches. To demonstrate its effectiveness, we've designed a sourcing Horus (online <query, ASINs> cache) experiment: applying it to improve <query, keyword> matches for a 1-year Amazon search historical engaged <query, ASINs> offline dataset to measure additional gains.

  • T1: the Pinpoint Token Match Engine was applied to ASINs with a review count of >= 10
  • T2: the Pinpoint Token Match Engine was applied to ASINs with a review count of >= 100

Overall, the Pinpoint Token Match Engine will potentially enhance recall for Torso/Tail queries and Torso/Tail ASINs. T2 aims to evaluate the influence of Tail ASINs.

The offline data set covers 277.5 million unique queries. Tested using 4 hours of Janus (ad) data, observed the distribution of new matches of sourcing in T1/T2 below:

What's Changing?

Build a rule engine solution

We introduce a rule engine that allows us to define segmentation and normalization rules in regular expression format. Rules are compiled to a single finite state machine that matches text very efficiently. To resolve the rule conflicts, we use left most longest rule. See the rule definition to normalize "num + unit pound" below. The rule engine excels in handling Apostrophe and Diacritics.

_define NUM [0-9]

_define POUND ((pound)|(pounds)|(lb)|(lbs))

_function FnNumUnit
  < ^ (NUM)+([./](NUM)+)? > --> MEASUREMENT
  < POUND > --> UNIT_POUND
_end

Mine and Select High Quality Rules

To support lemmatization, we applied Spacy lemmatization to tokens in Amazon ASIN titles. Selected 8k pairs from the top 13k labelings with token frequency > 400.

To support num+unit, num range, age and size segments, we exacted attributes such as size, age, num, unit from 1-year query log and summarized the top patterns.

Experimentation Design

There are multiple match types in Sponsored Product Search. The change is for Manual Targeting (MT) only. Based on match type definitions, the exact/phrase match rules are more restrictive, requiring close variations in terms and a specific sequence of orders. Meanwhile, broad match supports singulars, plurals, variations, synonyms, and related terms. We exclusively applied num_range match to broad match. The Pinpoint Token Match Engine (PPTME) was employed on a 1-year Amazon search historical engaged <query, ASINs> dataset, and additional MT candidates were published to the sourcing Horus - online <query, ASINs> cache.

  1. We performed a join operation on the engaged <query, ASINs> and ad data <ASIN, adId, tcId, keyword>, resulting in <query, ASIN, adId, tcId, keyword>.
  2. Both Lucene 5 tokenizer and the Pinpoint Token Match Engine (PPTME) were applied to query and keyword, producing Lucene_normalized_query, Lucene_normalized_keyword, PPTME_normalized_query, and PPTME_normalized_keyword.
  3. Candidates were selected if PPTME_normalized_query and PPTME_normalized_keyword could compose a new match.

Key Observations and Durable Learnings

  1. In T2, the Pinpoint Token Match Engine was applied to ASINs with a review count of >= 10, whereas in T3, it was applied to ASINs with a review count of >= 100. The distinction aimed to filter out certain COLD ASINs, optimizing for clicks. Improving matching alone can enhance visibility for certain tail/cold ASINs. The experiment also showed that T2 had more click lift.
  2. In addition to query and product match, SP search also involves matching constraints based on targeting clauses. To achieve optimal performance, we need to optimize both <query, ASIN> and <query, keyword> match simultaneously. Based on Sponsored Product Search Manual Targeting match types, the exact/phrase match rules are more restrictive, requiring close variations in terms and a specific sequence of orders. Meanwhile, broad match supports singulars, plurals, variations, synonyms, and related terms. Therefore, we need to carefully handle the match constraints. We need pinpoint matches for exact/phrase matches. For broad match, advertisers often use higher bids to target specific attributes. Relaxing to semantic matching could limit the options available to advertisers and potentially harm the ad system in the long term. When applying synonyms, it's important to avoid using them to match specific targeting clauses with general queries.
  3. Amazon's search teams invest significantly in recognizing named entities, extracting and normalizing attributes for queries and ASINs to enhance search quality. Leveraging these improvements in <query, keyword> matches can improve recall and ensure match accuracy. In query understanding, there are 8 head attributes (artist, title, size, brand, color, gender, material, and product_line) and over 1000 tail attributes (topic, department, target_audience, material, style, context_of_use, activity, occasion, event, genre). The attributes are identified as a text snippet. Efficient utilization requires additional normalization beyond existing efforts.
  4. We implemented a rule engine with customized regular expressions for segmentation and normalization. Token dictionaries are compiled to separate files and rules are compiled to a single finite state machine that matches text very efficiently. Rule conflicts are resolved using the leftmost longest rule. This weblab, addressing num/age segments and lemmatization/synonym techniques, aims to enhance <query, keyword> matching in an offline dataset. Expected to generate an annualized $108 million in SP revenue, it underscores the substantial potential of the rule engine.

Recommended Next Steps

  1. Extend rule engine solution to more techniques (context-based stop words, compounding,...) and segments/attributes (occasion, event, audience,...)
  2. Apply the Pinpoint Token Match engine to more offline data sets and online service
  3. Global launches to cover more languages
